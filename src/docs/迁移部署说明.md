# 迁移部署说明

本文面向“把本仓库迁移成你自己的站点”的使用者，重点说明静态站点、Vercel Serverless、环境变量、`API_BASE`、域名白名单和常见问题。

## 1. 先理解项目部署结构

本项目采用“静态站点和动态 API 分离”的部署方式:

1. GitHub Pages 负责托管 Astro 构建出的静态页面
2. Vercel 负责运行根目录 `api/` 下的 Serverless 函数
3. 前端通过 `src/config/json/api.json` 中的 `apiBase` 请求你的 Vercel API

所以迁移时不要把它理解成“只部署一个 Astro 站点”。

如果只部署 GitHub Pages 而没有完成 Vercel 配置，会出现这些情况:

- 页面可以打开
- 但登录、评论、留言、Firebase 配置注入、上传、人机验证等动态功能不可用

## 2. 迁移前准备

建议至少准备以下账号或服务:

- GitHub: 托管仓库与 GitHub Pages
- Vercel: 托管 Serverless API
- Firebase: Realtime Database 与前端 Firebase 配置

按需准备:

- GitHub OAuth App: GitHub 登录
- Cloudflare Turnstile: 人机验证
- Resend: 邮件验证码
- Cloudflare R2: 文件上传与图床
- 天地图、OpenWeather 等第三方 API Key

## 3. 创建你自己的仓库

1. 使用 GitHub 的 `Use this template` 或 `Fork`
2. 克隆到本地
3. 安装依赖

```bash
npm install
pip install -r requirements.txt
pip install -e .
```

4. 本地构建和预览

```bash
npm run build
quark serve
```

如果 `http://localhost:8000` 可正常访问，说明静态部分已经能跑通。

## 4. 先改掉与你站点直接相关的基础配置

### 4.1 站点信息

优先检查 `src/config/json/`:

- `index.json`: 个人信息、首页内容
- `index_config.json`: 首页板块显隐与顺序
- `friends.json`: 友链
- `manifest.json`: PWA
- `nav.json`: 站点标题、导航、登录相关配置
- `api.json`: API 基地址

### 4.2 图片资源

通常至少要替换 `assets/img/` 中这些资源:

- `bg.png`
- `logo_blue.png`
- `favicon.ico`
- `touxiang.png`

### 4.3 动态 API 域名

`src/config/json/api.json` 决定前端所有 `__API_BASE__` 占位符最终会被注入成什么地址。

当前结构:

```json
{
  "apiBase": "https://api.130923.xyz"
}
```

迁移后请改成你自己的 API 域名，例如:

```json
{
  "apiBase": "https://api.example.com"
}
```

注意:

- 必须写完整协议头，如 `https://`
- 不要写成 GitHub Pages 域名，应该写 Vercel 项目域名或你绑定到 Vercel 的自定义 API 域名
- 修改后需要重新执行 `npm run build`

## 5. GitHub Pages 静态部署

本仓库的静态页面构建产物在 `dist/`。推荐继续使用仓库中的 GitHub Pages 工作流来部署静态站点。

最低要求:

1. 仓库可正常 `npm install`
2. 仓库可正常 `npm run build`
3. GitHub Pages 指向构建产物发布流程

静态部署完成后，你应该先人工检查:

- 首页 `/`
- 文章列表 `/posts`
- 任意文章详情页
- 搜索页 `/search`

如果这些页面都正常，再继续接入动态能力。

## 6. Vercel Serverless 部署

### 6.1 为什么还需要 Vercel

本项目的以下能力依赖 `api/`:

- Firebase 公共配置注入: `/api/firebase-config`
- Firebase RTDB 代理: `/api/db`
- GitHub OAuth 用户信息获取: `/api/github-user`
- Turnstile 校验: `/api/turnstile-verify`
- 邮件验证码: `/api/email-send`、`/api/email-verify`
- R2 预签名上传: `/api/r2-presign`
- 管理员认证: `/api/admin-auth`、`/api/admin-verify`
- 动态 Key 注入: `/api/keys`

### 6.2 在 Vercel 导入项目

1. 登录 Vercel
2. 选择 `Add New Project` 或导入现有 Git 仓库
3. 选中你 fork/template 后的新仓库
4. Root Directory 保持仓库根目录
5. 保持 `vercel.json` 生效，不要把它改回普通 Astro 静态构建项目

本仓库的 `vercel.json` 已明确说明:

- `framework: null`
- `buildCommand` 只是输出“跳过静态构建”
- `outputDirectory: "."`

这是刻意设计的。Vercel 在这里不是用来构建 Astro 静态站点，而是只负责识别和运行 `api/*.js`。

所以如果你看到 Vercel 没有产出静态页面，不代表导入失败，反而通常是正常现象。

### 6.3 环境变量怎么加

在 Vercel 项目中打开 `Settings -> Environment Variables`，把下面你实际需要的变量补齐。

建议至少把“必需”和“你启用的功能对应项”全部添加到 `Production`。如果你还会使用 `Preview` 环境，也建议一并添加。

## 7. 环境变量清单

### 7.1 Firebase 相关

这组通常是启用动态功能的基础配置。

| 变量名 | 用途 |
|------|------|
| `FIREBASE_API_KEY` | 前端 Firebase 配置 |
| `FIREBASE_AUTH_DOMAIN` | 前端 Firebase 配置 |
| `FIREBASE_DATABASE_URL` | 前端配置与 RTDB Admin 读写 |
| `FIREBASE_PROJECT_ID` | 前端 Firebase 配置 |
| `FIREBASE_STORAGE_BUCKET` | 前端 Firebase 配置 |
| `FIREBASE_MESSAGING_SENDER_ID` | 前端 Firebase 配置 |
| `FIREBASE_APP_ID` | 前端 Firebase 配置 |
| `FIREBASE_MEASUREMENT_ID` | 前端 Firebase 配置，可选 |
| `FIREBASE_SERVICE_ACCOUNT` 或 `FIREBASE_SERVICE_ACCOUNT_JSON` | Serverless 端 Admin SDK 凭证 |

说明:

- `/api/firebase-config` 会把前 8 项注入前端
- `/api/db` 依赖 `FIREBASE_SERVICE_ACCOUNT` 和 `FIREBASE_DATABASE_URL`
- 若服务账号 JSON 直接复制到环境变量中，注意保留合法 JSON 格式
- 私钥里的换行通常可直接以转义形式存入，服务端会处理 `\\n`

### 7.2 GitHub OAuth

| 变量名 | 用途 |
|------|------|
| `GITHUB_CLIENT_ID` | GitHub OAuth |
| `GITHUB_CLIENT_SECRET` | GitHub OAuth |
| `GITHUB_API_KEY` | 前端通过 `/api/keys` 获取 GitHub API Token 时使用 |

额外要改:

- `src/config/json/nav.json` 中的 `login.githubUrl`

这个字段当前包含写死的 `client_id` 和 `redirect_uri`。迁移后你必须替换为你自己的 OAuth App 配置，否则登录会跳回原站或直接失败。

### 7.3 Turnstile

| 变量名 | 用途 |
|------|------|
| `TURNSTILE_SECRET_KEY` | 服务端验证 |

额外要改:

- `src/config/json/nav.json` 中的 `login.turnstileSiteKey`

服务端 Secret 和前端 Site Key 是两套值，缺一不可。

### 7.4 管理员认证

| 变量名 | 用途 |
|------|------|
| `ADMIN_PASSWORD_HASH` | 管理员密码哈希 |
| `ADMIN_SESSION_SECRET` | 管理员会话签名 |

### 7.5 邮件验证码

| 变量名 | 用途 |
|------|------|
| `RESEND_API_KEY` | Resend 发信 |
| `RESEND_FROM` | 发件人地址 |
| `EMAIL_TOKEN_SECRET` | 邮箱验证码签名 |

### 7.6 R2 上传

| 变量名 | 用途 |
|------|------|
| `R2_ACCESS_KEY_ID` | R2 S3 访问密钥 |
| `R2_SECRET_ACCESS_KEY` | R2 S3 密钥 |
| `R2_ENDPOINT` | R2 S3 兼容端点 |
| `R2_BUCKET` | 桶名 |
| `R2_PUBLIC_BASE_URL` | 公网访问前缀或自定义域名 |

### 7.7 第三方 Key 注入

| 变量名 | 用途 |
|------|------|
| `TIANDITU_KEY` | 天地图 |
| `OPENWEATHER_API_KEY` | OpenWeather |
| `QUARKCHAT_API_KEY` | QuarkChat |

不是所有站点都必须配置这些变量，只配置你实际会使用的功能即可。

## 8. 绑定你自己的 API 域名

推荐流程:

1. 先让 Vercel 项目成功部署
2. 记下它的默认域名，例如 `your-project.vercel.app`
3. 在 Vercel 的 `Domains` 中绑定自定义域名，例如 `api.example.com`
4. DNS 生效后，回到 `src/config/json/api.json`
5. 把 `apiBase` 改成 `https://api.example.com`
6. 重新构建并重新部署静态站点

这里一定要注意顺序:

- 先确认 Vercel API 已可访问
- 再把前端的 `apiBase` 切到新域名

否则你可能会把前端构建到了一个还没生效的 API 域名上。

## 9. 迁移时最容易漏改的地方

### 9.1 `nav.json` 中的 GitHub 登录地址

`src/config/json/nav.json` 里的:

- `login.githubUrl`
- `login.turnstileSiteKey`

都属于迁移时高频漏项。

其中 `login.githubUrl` 里不只是链接本身，还包含:

- `client_id`
- `redirect_uri`

如果你只在 Vercel 里配置了 `GITHUB_CLIENT_ID`/`GITHUB_CLIENT_SECRET`，但没有改 `nav.json`，前端仍会把用户带到旧的 OAuth 授权地址。

### 9.2 `api/*.js` 的来源白名单

多个 Serverless 文件都写了来源白名单，例如:

- `api/keys.js`
- `api/db.js`
- `api/github-user.js`
- `api/firebase-config.js`
- `api/turnstile-verify.js`
- `api/email-send.js`
- `api/email-verify.js`
- `api/r2-presign.js`
- `api/admin-auth.js`
- `api/admin-verify.js`

这些文件里目前仍包含原站域名，如:

- `lsqkk.github.io`
- `api.130923.xyz`
- `130923.xyz`

迁移后必须按你的实际域名替换，否则最常见的结果就是:

- 前端请求收到 `403 Forbidden`
- Firebase 配置无法注入
- GitHub OAuth 回调后拿不到用户信息
- Turnstile 明明前端渲染成功但服务端验证失败

如果你同时使用:

- `yourname.github.io`
- `www.example.com`
- `example.com`
- `api.example.com`

那么这些实际会发起请求的域名都要按需加入允许列表。

### 9.3 `db.js` 的允许路径

`api/db.js` 不是开放式数据库代理，它只允许固定根路径，例如:

- `presence`
- `user_activity`
- `chatrooms/lsqkk-lyb`
- `dynamic_posts`
- `post_annotations`
- `oj-discussions`
- `qb_users`
- `qb_email_index`
- `pic_upload_limits`
- `user_space_likes`
- `xjtu360`

如果你迁移后想新增新的 RTDB 根路径，但没有同步修改 `api/db.js` 的 `allowedRoots`，请求会报 `Invalid path`。

## 10. 推荐的迁移顺序

建议按这个顺序做，最不容易乱:

1. Fork 或模板化创建你的仓库
2. 本地安装依赖并跑通 `npm run build`
3. 修改 `src/config/json/` 和 `assets/img/`
4. 在 GitHub Pages 上先部署静态页面
5. 在 Vercel 导入同一个仓库，让 `api/` 可运行
6. 在 Vercel 中配置 Firebase 与其他功能所需环境变量
7. 绑定你的 Vercel API 域名
8. 修改 `src/config/json/api.json` 中的 `apiBase`
9. 修改 `src/config/json/nav.json` 中的 GitHub OAuth 和 Turnstile 配置
10. 全量检查 `api/*.js` 里的白名单域名
11. 重新构建并重新部署静态站点
12. 最后逐项验证动态功能

## 11. 建议逐项验证的功能

全部迁移完成后，建议最少检查:

- 首页是否能正常打开
- `/posts`、文章详情、`/search` 是否正常
- `/api/firebase-config` 是否能成功返回配置
- GitHub 登录是否能跳转并回到你自己的 `/auth.html`
- 留言、评论、动态等 RTDB 读写是否正常
- Turnstile 是否能通过服务端验证
- 若启用了 R2，文件上传是否成功

## 12. 常见问题

### 12.1 页面正常，但动态请求全是 403

优先检查:

1. `src/config/json/api.json` 的 `apiBase` 是否已改成你自己的域名
2. 对应 `api/*.js` 是否已加入你的站点域名白名单
3. 你访问站点的真实域名是否和白名单一致

例如你白名单只加了 `example.com`，但实际从 `www.example.com` 访问，也会失败。

### 12.2 Vercel 导入后看起来“没构建任何页面”

这是正常的。本项目的 `vercel.json` 就是为了让 Vercel 跳过静态构建，仅托管 `/api`。

### 12.3 GitHub 登录后回调异常

优先检查:

- `src/config/json/nav.json` 的 `login.githubUrl`
- GitHub OAuth App 的回调地址
- `/auth.html` 所在域名是否与你部署后的实际前端域名一致
- Vercel 是否设置了 `GITHUB_CLIENT_ID` 和 `GITHUB_CLIENT_SECRET`

### 12.4 Firebase 配置接口返回 403

优先检查 `api/firebase-config.js` 的 `allowedDomains` 是否已加入你的域名。

### 12.5 `Invalid FIREBASE_SERVICE_ACCOUNT JSON`

说明你存入 Vercel 的服务账号不是合法 JSON。最稳妥的方式是直接把完整服务账号 JSON 压成单行后保存到 `FIREBASE_SERVICE_ACCOUNT`。

### 12.6 本地可用，线上不可用

这是最常见的“本地域名已在白名单、线上域名没加”的情况。`localhost:8000` 能访问，不代表你正式域名也已被允许。

### 12.7 开启了 Preview，但预览环境功能失效

通常是因为环境变量只加到了 `Production`，没有同时加到 `Preview`。

## 13. 只想迁移静态部分怎么办

如果你只是想复用 Astro 页面、文章、样式和基础展示能力，可以只完成:

1. 本地构建
2. 配置修改
3. GitHub Pages 静态部署

但请接受这些功能会不可用:

- 登录
- 留言和评论
- 实时数据同步
- 动态上传
- Serverless 注入密钥
- 依赖 Firebase / Vercel API 的各类实验功能
