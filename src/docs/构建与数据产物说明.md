# 构建与数据产物说明

本文用于说明本仓库从源码到 `dist/` 的构建链路，以及若干“看起来像静态文件、实际由脚本生成”的数据产物。

## 1. 构建入口

标准构建命令：

```bash
npm run build
```

`package.json` 中的实际流程：

1. `prebuild`
2. `node scripts/generate-posts-json.mjs`
3. `node scripts/generate-rss.mjs`
4. `node scripts/sync-public.mjs`
5. `astro build`
6. `postbuild`
7. `node scripts/verify-dist.mjs`

`quark build` 只是对这套流程的封装；`quark build --mode artifact` 会在构建完成后把 `dist/` 额外复制到 `.quark-artifact/`。

补充：
- `quark updatelog` 会根据 Git 提交记录更新 `assets/data/log.json`
- 该命令默认会从现有 `log.md` 的最新日期开始刷新，自动将常见英文提交前缀整理成 `新增 -`、`更新 -`、`优化 -`、`修复 -` 格式

### 构建压缩
- CSS 压缩使用 LightningCSS（通过 `astro.config.mjs` 中 `vite.build.cssMinify: "lightningcss"` 配置）
- JS 压缩使用 esbuild（Vite 默认）
- 需安装 `lightningcss` 包：`npm install -D lightningcss`

## 2. 关键生成文件

### `posts/posts.json`
- 来源：`scripts/generate-posts-json.mjs`
- 输入：`posts/**/*.md`
- 用途：
  - 文章列表页
  - 首页最新文章
  - 文章封面数据
  - 搜索页文章元数据

脚本会自动提取：
- `title`
- `description`
- `date`
- `tags`
- `columns`
- `wordCount`
- `cover`

说明：
- 若旧 `posts.json` 中已有 `cover`，脚本会尽量复用
- 若没有，则会尝试请求远程随机封面地址

### `posts/rss.xml`
- 来源：`scripts/generate-rss.mjs`
- 输入：`posts/posts.json` + 对应 Markdown 正文
- 用途：RSS 订阅

### `public/`
- 来源：`scripts/sync-public.mjs`
- 说明：这是构建前同步出的公开静态目录，不建议直接手改

`sync-public` 主要会做这些事：
- 复制 `assets/` 到 `public/assets/`
- 复制 `src/config/json/` 到 `public/json/`
- 复制 `.nojekyll`、`robots.txt`、`BingSiteAuth.xml`、`posts/rss.xml`
- 把 `posts/posts.json` 复制为 `public/json/posts.json`
- 根据 `src/config/json/font.json` 和 `assets/fonts/` 生成 `public/assets/css/font-config.css`
- 生成兼容图片别名到 `public/image/`
- 将首页背景图配置注入 `shicha.js` / `shicha.css`
- 将 `api.json` 的 `apiBase` 注入所有包含 `__API_BASE__` 的文本文件

### `public/assets/css/font-config.css`
- 来源：`scripts/sync-public.mjs`
- 输入：`src/config/json/font.json` + `assets/fonts/*.woff(2)`
- 用途：给 `assets/css/basic.css` 提供全站字体变量 `--site-font-family`

`font.json` 的 `preferred` 可配置首选字体。`source` 支持：
- `.css` 字体样式地址，此时需在 `family` 指定该 CSS 内暴露的字体名
- `.woff` / `.woff2` 字体文件地址或本地公开路径，此时构建会自动写入 `@font-face`

若首选字体不可用，浏览器会按生成的字体栈继续回退：先使用 `assets/fonts/` 下扫描到的本地字体，再使用 `systemFallbacks` 中配置的常见系统字体。

### `assets/data/log.json`
- 来源：说明头 + `quark updatelog` 基于 Git 提交记录生成的日期分组内容
- 用途：
  - 首页”最近更新”
  - `/blog/log` 更新日志页

说明：
- 推荐通过 `quark updatelog` 更新，而不是手动复制 `git log` 结果
- 旧格式 `assets/md/log.md` 已不再使用

### `dist/json/site-pages.json`
- 来源：`astro.config.mjs` 中的 `apiBaseInjector` 插件
- 输入：
  - 构建完成后的 sitemap
  - 页面 `<title>`
  - 若干聚合 JSON：
    - `assets/pages/tool/tool.json`
    - `assets/pages/games/game.json`
    - `assets/pages/a/projects.json`
    - `assets/pages/blog/functions.json`
- 用途：
  - 站内搜索页
  - 页面索引补充

这意味着：新增页面如果只建了 Astro 路由、但没有在相应聚合 JSON 中登记，搜索页不一定能完整显示名称与入口。

## 3. `__API_BASE__` 注入机制

前端源码中允许写 `__API_BASE__` 占位符，而不是直接写死域名。

注入发生在两处：

1. `scripts/sync-public.mjs`
   - 处理 `public/` 下同步出的文本资源
2. `astro.config.mjs`
   - 处理最终 `dist/` 下文本资源

配置来源：

```json
src/config/json/api.json
```

当前结构示例：

```json
{
  "apiBase": "https://api.130923.xyz"
}
```

## 4. 与搜索相关的额外数据

搜索页 `src/pages/search/index.astro` 不只依赖 sitemap，还会额外准备：

- 文章全文索引：来自 `posts/posts.json` 与各 Markdown 正文
- 动态全文索引：来自 `assets/data/dt.json`

前端 `assets/js/site-search.js` 会综合读取：
- `/json/site-pages.json`
- `/json/posts.json`
- `sitemap*.xml`

所以改动这些内容时，需要一并验证 `/search`：
- 页面是否还能被搜索到
- 标题是否正确
- 文章标签 / 专栏 / 字数是否仍正常显示

## 5. 哪些文件通常不应手改

除非在修生成逻辑，否则不建议直接手动编辑：
- `public/**`
- `dist/**`
- `.quark-artifact/**`
- `posts/posts.json`
- `posts/rss.xml`
- `dist/json/site-pages.json`

正确做法通常是修改：
- `posts/**/*.md`
- `assets/**`
- `src/**`
- `src/config/json/**`
- `scripts/**`
- `astro.config.mjs`

## 6. 修改构建链路后的最低验证

至少执行：

```bash
npm run build
```

建议额外人工检查：
- 首页 `/`
- 文章列表 `/posts`
- 某篇文章详情页
- 动态页 `/blog/dt`
- 搜索页 `/search`
