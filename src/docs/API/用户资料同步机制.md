# 用户资料同步机制

## 架构

用户资料（昵称、头像、个性签名、背景图等）在三个层面存储：

| 层面 | 路径 | 用途 |
|------|------|------|
| Firebase RTDB | `user_activity/{uid}/profile` | 云端持久化，全站共享 |
| 浏览器 localStorage | `quark_user_profile` | 本地缓存，页面快速填充 |
| DOM 表单 | `#nicknameInput` 等表单元素 | 用户编辑界面 |

### 核心函数

**`assets/js/user-profile.js` — `syncProfile(profile)`**
全局唯一的 localStorage 写入入口。任何需要更新本地缓存的代码都调用此函数。它负责：
- 将 profile 写入 `localStorage.quark_user_profile`
- 分发 `quark-user-updated` 自定义事件
- **自动合并已有数据**（避免字段丢失，见下文）

**`assets/js/user-profile.js` — `buildProfile()` / `getProfile()`**
从 localStorage 构建当前用户 profile。读取多个来源：
- `localStorage.quark_user_profile`（完整缓存）
- `localStorage.nickname`、`postAnnoAvatarUrl` 等（旧版遗留 key）
- `localStorage.github_user` 或 `qb_user`（登录信息）

### 调用链

```text
保存资料（点击按钮）
  └─ account.js: saveProfile()
       ├─ Firebase .update(profile)       → 云端持久化
       └─ syncProfile(profile)             → 本地缓存

页面加载
  └─ nav.js 动态加载 user-activity.js
       └─ applyRemoteProfile(remoteProfile)
            └─ syncProfile(remoteProfile)  → 更新本地缓存
       └─ migrateLegacyUid()               → 旧 UID 数据迁移

  └─ account.js: loadRemoteProfile()
       └─ applyProfileToForm()             → 填充表单

  └─ account.js: fillStaticInfo()
       └─ getProfile() → buildProfile()   → 从 localStorage 填充表单

其他调用者（部分更新场景）
  ├─ login.js               → syncProfile({nickname, avatarUrl, ...})
  ├─ comment-input-shared.js → syncProfile({nickname, avatarUrl, ...})
  ├─ chat.js                → syncProfile({nickname, avatarUrl, ...})
  ├─ post-annotation.js     → syncProfile(profile)
  └─ live/js/index.js       → syncProfile({nickname, avatarUrl, ...})
```

## 已知问题与修复

### 问题：部分更新覆盖完整字段

**现象**：保存个性签名和背景图后，刷新一次能显示，第二次就没了。

**根因**：`user-activity.js` 在页面加载时调用 `syncProfile(remoteProfile)`，而 `remoteProfile` 是从 Firebase 读到的数据。如果 Firebase 中没有 `signature` 字段（例如旧数据未包含此字段，或被意外清除），`syncProfile` 会**直接用不完整的数据覆盖** localStorage 中完整的缓存，导致签名和背景图丢失。

类似的问题也发生在 `login.js`、`comment-input-shared.js` 等只传递部分字段的调用者。

**修复**：`syncProfile()` 内部自动与 localStorage 已有数据合并：

```javascript
function syncProfile(profile) {
    // 合并已有缓存，避免字段丢失
    const stored = readStoredProfile();
    if (stored && typeof stored === 'object') {
        const merged = {};
        for (const k in stored) merged[k] = stored[k];  // 旧数据做底
        for (const k in profile) merged[k] = profile[k]; // 新数据覆盖
        profile = merged;  // signature/backgroundImage 被保留
    }
    // ... 写入 localStorage
}
```

这样无论谁调用 `syncProfile({nickname, avatarUrl})`，已有字段都不会丢。

### 配套修复

- `account.js` `loadRemoteProfile()` — 读取 Firebase 后与 localStorage 合并再写入
- `user-activity.js` `applyRemoteProfile()` — 同样合并后再写入
- `user-activity.js` `migrateLegacyUid()` — 合并两个 profile 的所有字段，而非二选一

## 最佳实践

1. **修改 `syncProfile` 一处，全局生效**。无论新增什么字段，只需在 `syncProfile` 的合并逻辑中处理即可。
2. **调用 `syncProfile` 只传部分字段是安全的**——合并机制会保留其他字段。
3. **添加新字段时**：
   - 在 `buildProfile()` 中从 `readStoredProfile()` 读取
   - 在 `saveProfile()` 的 profile 对象中包含
   - `syncProfile` 的合并逻辑会自动处理新旧数据的兼容
4. **Firebase 与 localStorage 的 `updatedAt` 应保持一致**，避免 `user-activity.js` 在每次页面加载时误判"远程更新"而覆盖本地缓存。
